Androidmanifest 修复实践

发表于 2022-02-25 更新于 2022-03-01 分类于 Android逆向实践 阅读次数：

记录一次 Androidmanifest.xml 的修复实践过程。

工具准备

本次修复使用 apktool 作为 APK 反编译工具，并使用 010Editor 对 Androidmanifest.xml 文件进行十六进制修改。010Editor 中的官方 Androidmanifest 模板对于修改实践有很大的帮助。

修复过程

使用 jadx 查看 apk，发现其 Androidmanifest.xml 文件内容显示为乱码。使用 apktool 对 apk 进行反编译，反编译失败，输出日志如下：

解压缩出 Androidmanifest.xml 文件，使用 010Editor 打开，并使用 Androidmanifest 模板查看，发现其开头 magic number 值为 00080000h，该字段通常取值为 00080003h，该错误符合 apktool 错误日志。将其修改后重打包，再次使用 apktool 进行反编译，出现如下错误：

继续使用 010Editor 查看文件，检查其 stringChunk 结构，stringChunk 构成输下所示：

发现 StringCount 数值达到了 88674，遂检查其 strlem，发现总共存在 382 个 strlem 结构，且自 354 号开始均为乱码。将 StringCount 数值修改为 354（strlem 下标由 0 开始），移除后方指向乱码的偏移量。
继续检查，发现 strlem [41] 处也是一串乱码，为了减少工作量，并未采取删除整个乱码段的方式，而是使用了以下两种不同方法：

  1. 将 StringOffsets [41] 的值修改为其它正常字符串的偏移地址。
  2. 不修改 StringOffset 的值，而是将偏移指向的地址开始三字节修改为 000000h，前面 2 字节的 0 表示该字符串长度为 0，后面 2 字节表示字符串结束。通过这样的方式，将 strlem [41] 设置为空字符。

修改后的 Androidmanifest.xml 文件能够顺利反编译并显示正常内容。在 Androidmanifest 中插入了如此大量乱码的情况下，apk 还能正常安装并运行的原因有待学习。
